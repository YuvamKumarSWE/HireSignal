# HireSignal — Architecture & Technical Reference

## What is HireSignal?

HireSignal is an AI-powered mock interview simulator. A user signs in, pastes a job description, and gets interviewed by an AI — questions generated by Gemini, spoken aloud by ElevenLabs, answers submitted by text or voice. At the end, Gemini evaluates the full transcript and returns a hire probability, strengths, weaknesses, and a verdict. All results are saved to Firestore and visible on a profile page.

## Firebase

Firebase is Google's backend-as-a-service platform. HireSignal uses two Firebase products: **Auth** and **Firestore**.

### Firebase Auth

Firebase Auth handles identity. When a user clicks "Continue with Google":

1. The frontend calls `signInWithPopup(auth, googleProvider)` — Firebase opens a Google OAuth popup
2. The user approves access in Google's UI
3. Firebase verifies the OAuth response with Google and creates (or retrieves) a user account
4. Firebase issues the browser an **ID token** — a signed JWT (JSON Web Token)

The ID token is the core of the auth system. It's a base64-encoded string with three parts separated by dots:

```
eyJhbGciOiJSUzI1NiJ9.eyJ1aWQiOiJhYmMxMjMiLCJlbWFpbCI6InVzZXJAZ21haWwuY29tIn0.SIGNATURE
  [header]               [payload]                                                  [signature]
```

The **payload** contains claims — facts Firebase has signed and vouches for:
- `uid` — the user's unique Firebase ID (e.g. `"xK9mP2qRtVz..."`) — this is the stable identity across sign-ins
- `email` — the user's Google email
- `name` — display name
- `exp` — expiry timestamp (tokens expire after 1 hour)

The **signature** is created using Firebase's private key. Nobody can fake it without that key.

Firebase automatically refreshes the token before it expires. `auth.currentUser.getIdToken()` always returns a fresh, valid token.

### Firebase Admin SDK (backend)

The backend uses the **Firebase Admin SDK**, initialized with a **service account** — a JSON file containing a private key that proves the backend is a trusted server for your Firebase project.

```js
// backend/src/config/firebase.js
admin.initializeApp({
  credential: admin.credential.cert(JSON.parse(process.env.FIREBASE_SERVICE_ACCOUNT))
});
```

The Admin SDK can:
- **Verify ID tokens** — decode and cryptographically validate a token the frontend sent
- **Read/write Firestore** with full access (no security rules apply)
- **Create/delete users** (not used here, but available)

The client SDK (frontend) and Admin SDK (backend) are entirely separate packages with different capabilities. The client SDK is limited by Firestore Security Rules. The Admin SDK bypasses them entirely.

## How Tokens Flow

This is the complete auth flow from sign-in to a protected API call.

```
User                  Frontend                  Firebase               Backend
 │                       │                          │                     │
 │  Click Google sign-in │                          │                     │
 │──────────────────────>│                          │                     │
 │                       │  signInWithPopup()        │                     │
 │                       │─────────────────────────>│                     │
 │  Google OAuth popup   │                          │                     │
 │<──────────────────────│                          │                     │
 │  User approves        │                          │                     │
 │──────────────────────>│                          │                     │
 │                       │  Firebase verifies with Google                 │
 │                       │<─────────────────────────│                     │
 │                       │  Returns ID token (JWT)  │                     │
 │                       │<─────────────────────────│                     │
 │                       │                          │                     │
 │                       │  POST /api/user           │                     │
 │                       │  Authorization: Bearer <token>                 │
 │                       │────────────────────────────────────────────>   │
 │                       │                          │   verifyIdToken()   │
 │                       │                          │<────────────────────│
 │                       │                          │   {uid, email, ...} │
 │                       │                          │────────────────────>│
 │                       │                          │                     │  upsert user in Firestore
 │                       │  200 OK                  │                     │
 │                       │<────────────────────────────────────────────   │
```

### In code

**Frontend** — `api.js` attaches the token to every request automatically:

```js
// frontend/src/api.js
async function authHeaders() {
  const token = auth.currentUser
    ? await auth.currentUser.getIdToken()  // auto-refreshes if expired
    : null;
  return {
    'Content-Type': 'application/json',
    Authorization: `Bearer ${token}`
  };
}
```

**Backend** — `auth.middleware.js` intercepts every request to `/api/interview/*` and `/api/user/*`:

```js
// backend/src/middleware/auth.middleware.js
const token = req.headers.authorization.split('Bearer ')[1];
const decoded = await admin.auth().verifyIdToken(token);
req.user = { uid: decoded.uid, email: decoded.email, name: decoded.name };
next(); // only continues if token is valid
```

After the middleware runs, every controller has access to `req.user.uid` — the verified, trusted identity of whoever made the request.

### Why not just trust the userId from the request body?

Before this auth system was added, `Dashboard.jsx` was sending:

```js
userId: 'user_' + Date.now()  // a made-up string
```

The backend stored it without checking. Anyone could claim to be any user. With token verification, the `uid` comes from Firebase's signed token — it cannot be faked or guessed.

## Firestore

Firestore is a NoSQL document database. There are no tables or rows — instead:

- **Collection** — a named container for documents (like a folder)
- **Document** — a single record, identified by an ID, containing fields
- **Fields** — typed values: string, number, boolean, timestamp, array, map (nested object), or a reference to another document

### HireSignal's Data Model

#### `users/{uid}`

Created/updated on every sign-in via `POST /api/user`.

```
users/
  xK9mP2qRtVz.../          ← document ID = Firebase uid
    uid:         "xK9mP2qRtVz..."
    email:       "user@gmail.com"
    displayName: "Jane Smith"
    photoURL:    "https://..."
    createdAt:   Timestamp
```

#### `interviews/{sessionId}`

Created when an interview starts, updated when it ends.

```
interviews/
  session_1748234567_abc9x.../    ← document ID = sessionId
    userId:      "xK9mP2qRtVz..."  ← links back to user
    jobSpec:
      role:            "Frontend Engineer"
      level:           "Mid-level"
      jobDescription:  "..."
      company:         "Stripe"
    questions: [
      { id: "q1", question: "Tell me about yourself...", category: "introduction", expectedDuration: 120 },
      ...
    ]
    answers: []                   ← empty at start, filled on end
    evaluation: null              ← null until interview ends
    voice:
      id:          "JBFqnCBsd6RMkjVDRZzb"
      name:        "George"
      gender:      "male"
    startedAt:   Timestamp
    completedAt: null             ← null until interview ends
    status:      "active"         ← becomes "completed" on end
```

When the interview ends, the backend updates the document:

```
answers: [
  { questionId: "q1", question: "Tell me about...", answer: "I'm a...", answerType: "text", answeredAt: Timestamp },
  ...
]
evaluation:
  hire_probability: 0.78
  strengths: ["Clear communicator", ...]
  weaknesses: ["Lacks depth in X", ...]
  final_verdict: "Strong candidate..."
status:      "completed"
completedAt: Timestamp
```

### Querying past interviews

`ProfilePage.jsx` calls `GET /api/user/interviews`. The backend runs:

```js
db.collection('interviews')
  .where('userId', '==', uid)        // only this user's
  .where('status', '==', 'completed') // only finished ones
  .orderBy('startedAt', 'desc')      // newest first
  .get()
```

Firestore requires a **composite index** for queries that filter on multiple fields and sort. The first time this query runs, the backend logs an error with a direct URL to create the index in the Firebase Console — click it once and Firestore builds it automatically.

## Sessions

A session represents one complete interview. It has two lives: **in-memory** (during the interview) and **Firestore** (permanently).

### Why in-memory?

The interview is a stateful, fast-moving interaction — questions advance, answers accumulate, every request needs to know where you are. Reading and writing Firestore on every question/answer would be slow and wasteful. The in-memory Map is instant.

```js
// backend/src/services/interview.service.js
const interviewSessions = new Map();
//   sessionId → { questions, answers, currentQuestionIndex, status, ... }
```

### Session lifecycle

```
POST /api/interview/start
  → Gemini generates 5 questions
  → Random ElevenLabs voice selected
  → Session object created in the Map
  → Session document written to Firestore (status: "active", answers: [])
  → Returns { sessionId, totalQuestions, voice }

GET /api/interview/:sessionId/question
  → Reads session from Map
  → Returns the current question (by index)

POST /api/interview/:sessionId/answer  (or /answer/audio)
  → Adds answer to session.answers in Map
  → Increments currentQuestionIndex in Map
  → (Firestore not touched — wait until the end)

POST /api/interview/:sessionId/end
  → Reads all answers from Map
  → Builds transcript string
  → Sends transcript to Gemini for evaluation
  → Updates Firestore document: answers, evaluation, status: "completed", completedAt
  → Emits evaluation:complete via Socket.io
  → Returns evaluation + transcript
```

The session stays in the Map for the lifetime of the Node.js process. If the server restarts mid-interview, the in-memory session is lost — the user would need to start over. This is acceptable for a demo; a production system would use Redis for session state.

### The sessionId

Generated at interview start:

```js
const sessionId = `session_${Date.now()}_${Math.random().toString(36).substr(2, 9)}`;
// e.g. "session_1748234567890_k3mx9pq2a"
```

It serves three purposes simultaneously:
1. **Map key** — looks up the in-memory session object
2. **Firestore document ID** — the interview record's ID
3. **Socket.io room name** — all real-time events for this interview are scoped to this room
4. **URL parameter** — `/interview/session_1748234567890_k3mx9pq2a`

## Socket.io

Socket.io provides a persistent, bidirectional connection between the browser and the server — unlike HTTP which is request/response and then closed. This lets the server push data to the client at any time without the client polling.

### How the connection works

When `Interview.jsx` mounts, it opens a socket connection:

```js
// frontend/src/Interview.jsx
const socket = io(SOCKET_URL);            // opens WebSocket connection to backend
socket.emit('join:session', { sessionId }); // tells server "I'm in this interview"
```

On the backend, the server adds that socket to a **room** named after the sessionId:

```js
// backend/src/socket/index.js
socket.on('join:session', ({ sessionId }) => {
  socket.join(sessionId);  // socket is now in room "session_1748234..."
});
```

A **room** is a named group. When the backend does `io.to(sessionId).emit(...)`, the event goes to every socket in that room — in practice, just the one browser tab for that interview.

### Events used in HireSignal

| Direction | Event | Payload | When |
|---|---|---|---|
| Server → Client | `interview:timer` | `{ elapsed: 42 }` | Every second — drives the live timer in the UI |
| Server → Client | `transcription:status` | `{ status: 'processing' }` | When audio answer is being transcribed |
| Server → Client | `transcription:status` | `{ status: 'done', text: '...' }` | When transcription is complete |
| Server → Client | `evaluation:complete` | `{ evaluation, transcript }` | When `POST /end` finishes — navigates the user to results |
| Client → Server | `join:session` | `{ sessionId }` | On Interview mount |
| Client → Server | `leave:session` | `{ sessionId }` | On Interview unmount / tab close |

### Why push evaluation via Socket.io?

`POST /api/interview/:sessionId/end` calls Gemini for evaluation, which can take 3-10 seconds. The HTTP response and the Socket.io push both carry the evaluation — the HTTP response is the reliable fallback, the socket push is slightly faster because it fires as soon as Gemini responds, before the HTTP response headers are fully sent.

```js
// backend/src/controllers/interview.controller.js
const evaluation = await evaluateWithGemini({ transcript, role, level });
await endInterviewSession(sessionId, evaluation);         // save to Firestore
io.to(sessionId).emit('evaluation:complete', { evaluation, transcript }); // push via socket
res.json({ success: true, evaluation, transcript });      // also send via HTTP
```

### Timer

The backend keeps one timer per active session in a `Map<sessionId, setInterval>`. Every second it emits `interview:timer` to the session room. When the session room empties (user leaves or disconnects), the interval is cleared.

## Full End-to-End Flow

```
1. User visits /auth
   → clicks "Continue with Google"
   → Firebase opens Google OAuth popup
   → user approves
   → Firebase returns ID token + user object
   → frontend calls POST /api/user with token
   → backend verifies token, upserts user doc in Firestore
   → frontend navigates to /dashboard

2. User fills in role, level, job description → clicks "Begin Simulation"
   → frontend calls POST /api/interview/start with token
   → backend: verifies token → gets uid
   → backend: calls Gemini to generate 5 questions
   → backend: picks a random ElevenLabs voice
   → backend: creates session in Map
   → backend: creates interview doc in Firestore (status: active)
   → returns { sessionId, totalQuestions, voice }
   → frontend navigates to /interview/:sessionId

3. Interview.jsx mounts
   → opens Socket.io connection
   → emits join:session
   → backend starts 1-second timer, emitting interview:timer every tick
   → frontend calls GET /api/interview/:sessionId/question
   → backend returns question 1 from Map

4. For each question (text mode):
   → user types answer → submits
   → frontend calls POST /api/interview/:sessionId/answer
   → backend stores answer in Map, increments index
   → frontend fetches next question
   → repeats until hasMoreQuestions = false

4b. For each question (voice mode):
   → browser records audio (MediaRecorder API)
   → frontend uploads audio blob to POST /api/interview/:sessionId/answer/audio
   → backend emits transcription:status { processing } via socket
   → backend sends audio to ElevenLabs speech-to-text
   → backend emits transcription:status { done, text } via socket
   → backend stores transcribed text as answer in Map
   → frontend fetches next question

5. Last answer submitted → frontend calls POST /api/interview/:sessionId/end
   → backend builds transcript from all answers in Map
   → backend sends transcript + role + level to Gemini
   → Gemini returns { hire_probability, strengths, weaknesses, final_verdict }
   → backend updates Firestore: answers + evaluation + status: completed
   → backend emits evaluation:complete via socket
   → frontend receives it and navigates to /results/:sessionId

6. User visits /profile
   → frontend calls GET /api/user/interviews with token
   → backend queries Firestore: interviews where userId == uid, status == completed
   → returns array of past interviews
   → ProfilePage renders expandable cards with evaluation + Q&A
```
